# Lab Tracks App: Billing, Invoicing, Account Structure, and Admin Permissions Fixes

## Primary Issue: Invoicing and Billing Accuracy

The Lab Tracks app currently invoices cases as they are logged in, which is good, but the invoicing logic is not correctly respecting pricing tiers or individual account pricing.

### Example Problem

When a case is logged for:

**Tooth #3 Crown — Zirconia**

The system labels it as **Zirconia** and bills **$250**, even though that account does not have a $250 price point assigned to their pricing tier.

This means invoices are not correctly pulling from:

* The assigned pricing tier
* The individual account’s custom pricing
* Any edited or updated pricing values

## Required Fix

The invoicing system needs to always calculate pricing from the correct source, in this order:

1. Individual doctor/account-specific pricing
2. Practice or group-specific pricing
3. Assigned pricing tier
4. Default product/service pricing only if no other price exists

Any edit made to pricing tiers or individual account prices must immediately reflect on new invoices and, where appropriate, existing editable invoices.

---

# Pricing Management Requirements

## Pricing Tiers

Admins should be able to:

* Create pricing tiers
* Edit pricing tiers
* Assign pricing tiers to doctors, practices, or groups
* Confirm that invoice line items reflect the correct tier price

## Individual Account Pricing

Admins should be able to:

* Override pricing for an individual doctor/account
* Edit custom prices
* Confirm those prices are used on invoices
* Prevent the system from falling back to incorrect default pricing

# Doctor, Practice, Group, and Location Structure

The app needs a cleaner structure for organizing doctors, practices, groups, and office locations.

## Example Structure

**CWD Dental Group**

Locations:

* Centerville Road office
* Cary Forest Parkway office, Tallahassee

Doctors:

* Dr. Couch
* Dr. Woods
* Dr. Mullen
* Dr. Dalton

## Required Behavior

Each doctor should be tracked and billed individually.

For example:

* Dr. Woods’ cases should be billed to Dr. Woods
* Dr. Dalton’s cases should be billed to Dr. Dalton
* Dr. Couch’s cases should be billed to Dr. Couch

However, statements and invoices may need to be sent to one shared office manager for the group.

## Needed Data Relationships

The app should support:

* Group: CWD Dental Group
* Practice locations under that group
* Individual doctors under the group and/or location
* Billing contact for the group
* Individual billing records per doctor
* Shared statement delivery options

---

# Doctor Matching / AI Prescription Reading

When AI reads a prescription, doctor names may appear in different formats.

Examples:

* Dr. Jake Woods
* Dr. Woods
* Woods

The app should detect similar names, especially when paired with a similar address or practice, and ask the user to confirm whether this is the same doctor.

## Required Confirmation Flow

When a possible duplicate doctor is detected, show a prompt:

**“This doctor looks similar to an existing doctor: Dr. Jake Woods at CWD Dental Group. Is this the same doctor?”**

Options:

* Yes, use existing doctor
* No, create new doctor
* Edit details before saving

If this was already added recently, verify that it is working correctly. If not, this needs to be implemented.

# Invoice Editing Improvements

Invoice editing is inconsistent depending on where the invoice is accessed.

## Current Issue

Admins can access invoices from:

1. Quick reference invoice button on the case summary
2. Admin Hub → Financial → Invoices

The problem is that the case summary invoice editor only allows limited edits, while the Admin Hub invoice editor allows more complete editing.

## Required Fix

Invoice editing permissions and available fields should be the same no matter where the invoice is opened from.

Admins should be able to edit every invoice detail from either location, including:

* Doctor/account
* Practice/group
* Billing address
* Line items
* Product/service category
* Tooth number
* Material
* Quantity
* Unit price
* Discounts
* Adjustments
* Notes
* Tax, if applicable
* Invoice status
* Payment status
* Statement grouping

---

# Admin Permissions Issue

Admin permissions are currently tied too heavily to entering the Admin Master Hub.

## Current Problem

Even when logged into an admin account, certain admin features are unavailable unless the user first enters the Admin Master Hub using Face ID.

Example:

The quick reference invoice button on the case summary may not work unless the admin has already entered the Admin Master Hub.

## Required Behavior

If a user is logged in with an admin account, they should automatically have access to all admin features allowed by their role.

They should not have to enter the Admin Master Hub first to “activate” those permissions.

## Required Fix

Admin permissions should be checked globally based on the logged-in user’s role, not based on whether they accessed the Admin Master Hub during the session.

# Billing and Statement Flow Goals

The billing system should support:

* Accurate invoice generation when cases are logged
* Correct doctor-specific billing
* Group-level statement delivery
* Editable invoices from all admin access points
* Reliable pricing tier and custom pricing logic
* Clear doctor/practice/group relationships
* Duplicate doctor detection from AI prescription reading
* Global admin permissions once logged in
