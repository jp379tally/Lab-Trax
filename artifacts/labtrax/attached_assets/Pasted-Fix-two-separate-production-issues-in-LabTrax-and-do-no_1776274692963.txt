Fix two separate production issues in LabTrax and do not treat them as one bug.

Issue 1: join request rejection bug

Current behavior:
Rejecting a join request sometimes throws Postgres error 23505:
duplicate key value violates unique constraint "join_requests_lab_user_status_unique"
Key (lab_id, user_id, status) = (..., ..., rejected) already exists.

Root cause:
The schema currently enforces uniqueness on (lab_id, user_id, status), which is too strict for a workflow that keeps historical rows for approved/rejected/cancelled requests.

Required fix:
1. Change the uniqueness rule so only one pending join request can exist per (lab_id, user_id).
2. Allow multiple historical approved/rejected/cancelled rows.
3. Make approve/reject endpoints idempotent:
   - if already rejected, return success instead of error
   - if already approved, return a controlled 409 or clear no-op response
4. Safely deduplicate existing rows before applying the new index/constraint.
5. Update UI refresh/click handling so repeated clicks do not create duplicate state transitions.

Preferred DB solution:
- Drop the current unique constraint/index on (lab_id, user_id, status)
- Replace it with a partial unique index enforcing only one pending request per (lab_id, user_id)

Desired endpoint behavior when rejecting:
- fetch row by id
- if not found -> 404
- if status === 'rejected' -> return ok true / already rejected
- if status !== 'pending' -> return controlled validation response
- otherwise update to rejected

Also test:
- reject same request twice
- approve same request twice
- reject after approve
- approve after reject
- rapid repeated button taps from UI
