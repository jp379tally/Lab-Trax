In the full app, the signup flow uses the real backend org system: LoginScreen.tsx (around lines 741–795) searches labs with GET /api/labs/groups and submits joinOrganizationId. But the Settings / Home / Notifications flows are still using device-local storage: lib/app-context.tsx loads join requests and invites from AsyncStorage around lines 409–417, then saves them back to AsyncStorage around lines 1288–1407; app/settings.tsx uses that local flow around lines 840–906; app/(tabs)/notifications.tsx reads groupJoinRequests and labInvitations around lines 103–123. That is why device A can “send” something and device B never sees it. Your uploaded backend already has real organization, join-request, and invite endpoints, and the project notes say this is a starter that still needs wiring.
I also found a second hard bug: when an admin accepts a join request or invite, the frontend tries to update the other user’s profile through PUT /api/auth/users/:id/profile in lib/app-context.tsx around lines 1329–1438. But the backend route in server/routes/auth.ts only allows a user to update their own profile around lines 377–385. So that request gets a 403, and the frontend swallows the error. That means the UI can say “accepted” while the backend never actually links the other user.
That also explains your new “create lab” issue. In Settings, “My Lab” is shown from currentUserData.practiceName in app/settings.tsx around lines 344–347, and lab search only indexes registeredUsers where userType === "lab" && role === "admin" && practiceName around lines 840–846. So if your create-lab change created an organization but did not update the user profile string that the UI depends on, the lab won’t show under My Lab. And even if the org exists, search can still miss it, because the UI is searching users, not organizations. In your server code, organization creation is a real backend action with owner membership.
The quickest fix for a same-day publish is this: stop using sendGroupJoinRequest and the local invite arrays for anything related to labs. In Settings, search labs from /api/labs/groups, not from registeredUsers. Then submit join requests with the real backend route: POST /api/organizations/:organizationId/join-requests. The backend route already exists in the app server at server/routes/organizations.ts around lines 319–345, and the public lab list endpoint exists at server/routes.ts around lines 88–140.
