# Lab Tracks Provider Portal Development Plan

We now need to begin building the **provider portal side** of Lab Tracks.

So far, most work has focused on the **lab-side environment**, including:

* Lab member login
* Closed members-only lab environment
* Shared lab data across users/devices
* Case entry
* Billing
* Invoicing
* Admin workflows

Now we need a separate but connected **provider portal**, where doctors/providers can log in and view only their own information.

---

# Provider Portal Goal

The provider portal should allow each provider to securely access:

* Their cases
* Their invoices
* Their statements
* Their account information
* Their practice/group information
* Their case status updates
* Their delivery or pickup information, if applicable

The provider portal should use the same security systems as the lab side, including:

* Same login system
* Same two-factor authentication
* Same Face ID / face recognition flow
* Same database backend

However, providers should only see data connected to their specific provider account.

---

# Critical Provider Data Separation Requirement

Providers must never see another provider’s cases by accident.

Example:

There may be two providers named **Dr. Woods**:

* Dr. Woods in Arizona
* Dr. Woods in California

Each Dr. Woods must only see their own cases, invoices, and statements.

The system should not rely only on doctor name matching.

Provider access must be tied to a unique provider account ID.

---

# Provider Account Number System

Each provider should receive an automatically generated account number.

## Account Number Format

The account number should use:

1. Provider first initial
2. Provider last initial
3. Numerical portion of the provider’s address

Example:

**Jacob Woods**
**123 Happy Street**

Account number:

**JW123**

---

# Duplicate Account Number Handling

If another provider has the same initials and same street number, the system should add a letter suffix.

Example:

**Jacob Woods**
123 Happy Street
Account number: **JW123**

**Julia Wilson**
123 Happy Street
Account number: **JW123B**

If another duplicate exists:

**James Walker**
123 Happy Street
Account number: **JW123C**

The first provider gets no suffix. Additional providers get suffixes in order:

* JW123
* JW123B
* JW123C
* JW123D

---

# Automatic Provider Account ID Generation

Whenever a new provider is created or detected, Lab Tracks should automatically generate a provider account ID.

This should happen when:

* A new provider is manually added
* A new provider is detected from an AI-read prescription
* A new case is entered for a provider not already in the system
* A provider is created through admin tools

The generated account ID should then become the permanent unique identifier for that provider.

---

# Important Recommendation

The visible account number, such as **JW123**, should be used as a friendly account number.

But internally, the database should also create a hidden unique provider ID, such as:

**provider_uuid**

This prevents problems if:

* A provider changes address
* A provider changes name
* Two providers accidentally generate similar account numbers
* An account number needs to be edited later

So the system should use:

* **provider_uuid** for internal database relationships
* **provider_account_number** for human-facing account lookup

---

# Provider Portal Account Linking

When a provider creates a portal account, the system needs to connect that login to the correct provider record.

## Suggested Provider Signup Flow

When a provider signs up, ask for:

* First name
* Last name
* Email
* Phone number
* Practice name
* Street address
* Account number, if they already have one

Then the system should search existing provider records using:

* Email match
* Phone number match
* Address match
* Practice/group match
* Provider account number match
* Similar name match

---

# Existing Provider Match Flow

If the system finds a likely match, show a confirmation screen.

Example:

**“We found an existing provider account that may belong to you.”**

Matched account:

**Dr. Jacob Woods**
**CWD Dental Group**
**123 Happy Street**
**Account Number: JW123**

Options:

* Yes, this is me
* No, this is not me
* Request admin review

---

# Admin Approval for Provider Linking

For security, providers should not automatically gain access to cases just because they enter matching information.

Recommended flow:

1. Provider creates portal login
2. System searches for matching provider account
3. Provider requests connection
4. Lab admin receives approval request
5. Admin approves or rejects the connection
6. Once approved, provider can see only their linked data

This prevents someone from guessing an account number and accessing private case information.

---

# Provider Portal Data Rules

Once a provider account is linked, the provider should only see data where:

* case.provider_id matches their provider_uuid
* invoice.provider_id matches their provider_uuid
* statement.provider_id matches their provider_uuid
* practice/group access is explicitly granted

A provider should not see cases simply because the name is similar.

---

# Group / Practice Access Rules

Some providers may belong to a larger group, such as:

**CWD Dental Group**

Doctors:

* Dr. Couch
* Dr. Woods
* Dr. Mullen
* Dr. Dalton

Locations:

* Centerville Road
* Cary Forest Parkway

Each doctor should have their own provider account.

However, an office manager may need access to multiple providers under the group.

## Suggested Roles

Provider portal should support roles:

### Individual Provider

Can see only their own cases, invoices, and statements.

### Office Manager

Can see approved providers within the practice or group.

### Group Admin

Can see all providers, locations, invoices, and statements under the group.

### Lab Admin

Can manage all provider relationships from the lab side.

---

# Provider Portal Permissions

Provider permissions should be based on explicit account linking, not just login status.

The portal should support:

* Provider login
* Provider account linking
* Admin approval
* Role-based access
* Multi-location support
* Group-level access
* Individual doctor-level billing visibility

---

# AI Prescription Provider Matching

When AI reads a prescription, it may detect provider names in different formats.

Examples:

* Dr. Jake Woods
* Dr. Jacob Woods
* Dr. Woods
* Woods

The system should compare:

* Name similarity
* Address
* Phone number
* Practice name
* Existing provider account number

If there is a likely match, the system should ask the lab user:

**“This provider looks similar to an existing provider. Is this the same provider?”**

Options:

* Yes, use existing provider
* No, create new provider
* Edit before saving

# Suggested Technical Structure

## Provider Table

Each provider should have:

* provider_uuid
* provider_account_number
* first_name
* last_name
* display_name
* email
* phone
* address
* practice_id
* group_id
* location_id
* active_status

## Provider Portal User Table

Each portal user should have:

* user_uuid
* email
* phone
* authentication_method
* two_factor_enabled
* face_id_enabled
* provider_uuid, if linked
* role
* approval_status

## Provider Account Link Table

Use this table to track connection requests.

Fields:

* link_request_id
* portal_user_uuid
* requested_provider_uuid
* match_confidence
* requested_at
* approved_by_admin
* approved_at
* status

Statuses:

* pending
* approved
* rejected
* needs_review

When a provider is created, check for similar existing providers.

Match based on:

* Name
* Address
* Phone
* Email
* Practice/group

Ask the user before creating duplicates.

## Phase 3: Provider Portal Signup

Create provider portal signup/login flow using the same authentication system as the lab side.

Providers should be able to request access to an existing provider account.

---

## Phase 4: Admin Approval

Create an admin review screen where lab admins can approve or reject provider portal account linking requests.

---

## Phase 5: Provider Dashboard

After approval, the provider should see a portal dashboard with:

* Active cases
* Completed cases
* Invoices
* Statements
* Account information
* Messages or notes from the lab

---

## Phase 6: Office Manager / Group Access

Add support for office managers who can access multiple doctors under a practice or group.

This access must be granted by lab admin approval.

---

# Final Recommendation

Do not rely on doctor names alone.

Use this structure:

* Human-facing account number: **JW123**
* Internal permanent provider ID: **provider_uuid**
* Portal login ID: **user_uuid**
* Admin-approved link between login and provider account

This gives us a safe provider portal where doctors see only their own cases, while still allowing office managers and group admins to access multiple providers when approved.
