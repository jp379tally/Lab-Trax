# Labtrax Financial Register, Bank Sync, Cash Flow, and Memorized Transactions Feature Spec

## Goal

Add a financial module to Labtrax that works like a QuickBooks-style bank/checkbook register. The user should be able to:

1. View and manage a checking account register.
2. Reconcile the bank account through the website.
3. Connect a real bank account so transactions are imported automatically.
4. Show pending and actual cash flow by month.
5. Automatically create recurring or “memorized” transactions.
6. Display the financial dashboard in a familiar QuickBooks-like register layout.

Use a bank connection provider such as Plaid. Plaid supports bank transaction syncing and webhooks, so the app can receive updates when transactions change instead of manually entering everything. ([Plaid][1]) Replit scheduled deployments can be used for monthly jobs like generating recurring transactions. ([Replit Docs][2])

---

## Main Features

### 1. Bank / Checkbook Register

Create a register screen similar to a QuickBooks bank register.

Each row should show:

* Date
* Transaction type
* Check number or reference number
* Payee / vendor
* Memo
* Category / account
* Debit
* Credit / deposit
* Cleared status
* Reconciled status
* Running balance
* Source: manual, bank import, recurring, adjustment

The register should allow:

* Add manual transaction
* Edit transaction
* Delete or void transaction
* Mark as cleared
* Mark as reconciled
* Match imported bank transaction to existing manual or recurring transaction
* Filter by date, status, payee, category, or amount
* Search transactions
* Export CSV

---

### 2. Bank Connection

Add a bank connection workflow.

Recommended flow:

1. User clicks “Connect Bank Account.”
2. App opens bank-linking flow using Plaid Link.
3. Store the bank item and account IDs securely.
4. Store access tokens only in server-side environment variables or encrypted database fields.
5. Import historical transactions.
6. Listen for Plaid transaction webhooks.
7. When new bank activity is available, call the transaction sync endpoint.
8. Insert new transactions into the register.
9. Update modified or removed bank transactions.

Important: Do not store bank login credentials. The app should only store secure tokens from the provider.

Bank sync behavior:

* Imported bank transactions should default to `cleared = true`.
* Imported transactions should not automatically be `reconciled`.
* If an imported transaction matches a projected recurring transaction, mark the recurring transaction as matched.
* If no match exists, create a new register entry.

Matching rules:

* Same amount
* Similar date, within configurable range such as 3 to 5 days
* Similar payee/vendor name
* Same bank account
* Same transaction direction: debit or credit

---

### 3. Reconciliation

Create a reconciliation screen where the user can reconcile the checking account.

Fields:

* Bank account
* Statement start date
* Statement end date
* Beginning balance
* Ending statement balance
* Cleared deposits
* Cleared payments
* Calculated cleared balance
* Difference

Workflow:

1. User enters statement ending balance and date.
2. App shows all unreconciled transactions up to that date.
3. User checks off transactions that appear on the statement.
4. App calculates the difference.
5. If difference is zero, allow “Finish Reconciliation.”
6. Mark selected transactions as reconciled.
7. Save reconciliation history.

---

### 4. Cash Flow Analysis

Create a cash flow dashboard by month.

Formula:

Revenue for month - Expenses for month = Net cash flow

Show:

* Total revenue
* Total expenses
* Net cash flow
* Starting cash balance
* Ending cash balance
* Projected future balance
* Actual vs projected cash flow
* Category breakdown
* Payroll forecast
* Loan/payment forecast
* Upcoming large expenses

Views:

* Current month
* Prior month
* Next 30 days
* Next 60 days
* Next 90 days
* Custom date range

Use actual posted transactions plus projected recurring transactions.

---

### 5. Memorized / Recurring Transactions

Build a recurring transaction engine similar to QuickBooks memorized transactions. QuickBooks lets users memorize repeating transactions and optionally enter them automatically. ([QuickBooks][3])

Recurring transaction examples:

* Payroll on the 1st and 15th
* Car payment on the 12th
* Loan payment monthly
* Rent
* Insurance
* Utilities
* Subscriptions
* Regular vendor payments

Each recurring rule should include:

* Name
* Payee
* Amount or estimated amount
* Transaction type: debit or credit
* Category/account
* Memo
* Start date
* End date, optional
* Frequency: weekly, biweekly, monthly, semi-monthly, quarterly, yearly
* Day of month
* Auto-create setting
* Estimate from last matching transaction setting
* Active/inactive status

Automation behavior:

* On the first day of each month, generate projected transactions for the rest of the month.
* Example: if payroll usually occurs on the 1st and 15th, create projected payroll entries for both dates.
* If the amount varies, estimate based on the last actual transaction or average of the last 3 matching transactions.
* When the real bank transaction imports, match it against the projected transaction.
* Replace the projected amount with the actual amount.
* Mark it as cleared once imported from the bank.
