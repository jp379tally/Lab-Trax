### Replit Build Brief

**Objective**
Fix batch locate so moving multiple cases from one location to another actually updates the cases and refreshes correctly on both mobile and web.

**User-visible problem**
The location picker now allows selecting one location at a time, but when the user batch locates four cases from **Received** to **Design**, the cases do not show the updated location in either the mobile app or the web version.

**Likely root cause**
The UI selection bug was only the first half. The submit/update path is now failing in one of these ways:

* the batch locate request is not sending the selected destination `locationId`
* the request sends case IDs but the backend ignores or rejects the destination location
* backend updates a different field than the lists read from
* mutation succeeds but mobile/web caches are not invalidated/refetched
* optimistic UI updates are missing or immediately overwritten by stale query data
* the cases are moved in the database but the current location/status filters still show stale results

### Debug Analysis

**Primary root-cause hypothesis**
The picker state is fixed, but the batch locate mutation is not correctly updating the persisted case location, or the app is not refetching/invalidation cached case data after the update.

**Supporting evidence**
The same stale result appears on both mobile and web. That usually means either the backend/database did not actually update, or both clients are reading cached/stale query data from the same source.

**Most likely failure points**

* `selectedLocationId` is correct in the UI but not included in the submit payload.
* Batch API expects a different field name, such as `newLocationId`, `location`, `statusId`, or `workflowStepId`.
* Backend updates only the first case or zero cases.
* Backend updates `locationId`, but lists display `currentLocation`, `status`, or another derived field.
* React Query/SWR cache is not invalidated after mutation.
* The current filter still shows old data because local state was not reconciled.

**Why the recommended approach is best**
This needs to be fixed at the persistence and cache boundary, not with another UI-only patch. If the backend updates the canonical case location and both clients refetch that same field, mobile and web will stay consistent.
