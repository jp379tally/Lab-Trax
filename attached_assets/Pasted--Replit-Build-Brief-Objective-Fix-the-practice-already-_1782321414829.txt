### Replit Build Brief

**Objective**
Fix the “practice already exists” blocker so LabTrax only rejects a new practice when a real matching active practice exists, and so the UI shows the user exactly what record is causing the conflict.

**User-visible problem**
The Customer Center says:

```text
A practice with this name already exists. Select existing practice instead.
```

But the visible practice list in the background does not show **Dr. Susan Byrne** or a matching practice. The user cannot create the practice and has no way to resolve the conflict.

**Likely root cause**
The duplicate check is broader than the visible Customer Center list. It is probably matching one of these:

* a soft-deleted / archived / inactive practice
* a hidden practice not returned by the current list filter
* a doctor/provider record being treated as a practice duplicate
* normalized name logic matching incorrectly
* an auto-generated account number conflict derived from `Dr. Susan Byrne`
* duplicate validation using `displayName`, `legalName`, `primaryDoctorName`, or account number inconsistently

**Implementation plan**

1. Find the backend duplicate check that returns “A practice with this name already exists.”
2. Confirm which field is triggering the duplicate:

   * `legalName`
   * `displayName`
   * `primaryDoctorName`
   * generated `accountNumber`
   * doctor/provider name
   * email/phone/address
3. Change duplicate detection so practice name conflicts only compare against **active practices that are visible/selectable in Customer Center**, unless the UI intentionally supports restoring archived practices.
4. Do not let a doctor/provider record block creation of a practice unless it belongs to an existing active practice.
5. If a soft-deleted or archived practice matches, return a specific message:

   ```text
   This practice exists but is archived. Restore it or create with a different name.
   ```
6. If an active duplicate exists, return the matched practice ID/name so the UI can show a link or select it.
7. If the conflict is actually account-number generation, change the error message to say account number already exists and generate a unique suffix instead of blocking on the name.
8. Make the Customer Center list/search able to reveal the matching record when duplicate validation says one exists.
9. Add tests for active duplicate, archived duplicate, doctor-name-only match, and account-number collision.

**Guardrails**

* Do not remove duplicate protection entirely.
* Do not allow two active practices with the same normalized legal/display name unless that is already permitted by business rules.
* Do not treat doctors as practices.
* Do not hide the conflicting record from the user.
* Avoid broad matching like `contains`, fuzzy matching, or comparing only first/last doctor name for practice uniqueness.

**Acceptance criteria**

* Creating **Dr. Susan Byrne** succeeds if no active visible practice with that exact normalized practice name exists.
* If an active duplicate exists, the error identifies the existing practice and the user can select/open it.
* If only an archived/deleted practice exists, the UI gives a restore/select option or allows recreation based on business rules.
* A doctor named Dr. Susan Byrne under another practice does not block creating a new practice named Dr. Susan Byrne unless the app explicitly treats provider names as globally unique.
* Account-number collisions are handled by generating a unique account number or showing the correct account-number error.
* Customer Center search for “Susan Byrne” finds the record if duplicate validation says it exists.

### Debug Analysis

**Primary root-cause hypothesis**
The duplicate check is querying a wider data set than the Customer Center list displays, or it is checking the wrong entity. The app may be finding a hidden/archived record, a provider/doctor row, or an account-number collision, then reporting it as a practice-name duplicate.

**Supporting evidence**
The screenshot shows the modal error while the visible Customer Center list includes practices like Bastien Dental Care, Blissful Dental Spa, Brittney K Craig DMD, CWD Dental Group, First Care Dental Associates, Russell Bartow Rainey DMD, and SDR1. It does not show Dr. Susan Byrne.

**Alternate hypotheses worth checking**

* The practice exists on another page of results, but search/list is not being refreshed or searched correctly.
* Name normalization is over-aggressive, causing unrelated names to collide.
* `primaryDoctorName` is being used as a globally unique practice key.
* A deleted/archived practice remains in the database with a unique index.
* Auto account number generation creates a duplicate but maps the error to the wrong message.

**Why the recommended fix is best**
The app needs one consistent source of truth: if creation is blocked because a practice exists, the user must be able to see and select that practice. If the matching record is hidden, archived, or a doctor row, the error should not block creation as a visible active practice duplicate.
