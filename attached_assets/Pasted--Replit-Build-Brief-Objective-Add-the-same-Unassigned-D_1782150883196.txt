### Replit Build Brief

**Objective**
Add the same **Unassigned Documents** window to the **top-left of the mobile layout** that already exists on desktop, and connect mobile picture capture into that window.

**User-visible problem**
On mobile, the user cannot quickly take a picture and have it appear in the Unassigned Documents area the way desktop users can manage unassigned documents.

**Likely root cause**
The desktop layout has an Unassigned Documents component/queue, but the mobile layout either hides it, moves it out of reach, or uses a different upload flow that does not route newly captured images into the same unassigned-document state.

**Exact areas to inspect/change**

Search the app for:

* `Unassigned Documents`
* `unassigned`
* `documents`
* `upload`
* `dropzone`
* `Add file`
* `Add picture`
* `camera`
* mobile layout components or responsive breakpoints

Likely surfaces:

* The desktop dashboard/layout component that renders the Unassigned Documents window
* The mobile dashboard/layout component
* The file upload or image upload component
* The document queue/state/store/API used for unassigned documents
* Any modal/dialog system used for renaming files

**Implementation plan**

1. Reuse the existing desktop Unassigned Documents component on mobile instead of creating a separate duplicate flow.
2. Place it at the top-left of the mobile screen, above or beside the main mobile work area depending on the current responsive layout.
3. Add a mobile upload/take-picture target inside or near that window.
4. Use a hidden file input for camera capture:

```tsx
<input
  type="file"
  accept="image/*"
  capture="environment"
/>
```

5. When the user taps the “add a file or picture” area on mobile, trigger that input.
6. After the camera returns an image, show a rename prompt/modal before finalizing the upload.
7. If the user renames it, use the new filename while preserving the image extension.
8. If the user skips renaming, keep the default generated filename.
9. Add the captured image into the same Unassigned Documents list/queue used by desktop.
10. Make sure existing desktop behavior is unchanged.

**Guardrails**

* Do not build a separate mobile-only document system.
* Reuse the existing unassigned-document state and upload API.
* Preserve file extensions when renaming.
* Sanitize filenames to avoid slashes, special path characters, or blank names.
* Handle cancel cases cleanly:

  * user cancels camera
  * user cancels rename
  * upload fails
* On mobile browsers, `capture="environment"` usually opens the rear camera, but some browsers may show a camera/photo picker. That is normal browser behavior.

**Acceptance criteria**

* On mobile, an Unassigned Documents window is visible at the top-left.
* Tapping the add file/picture area opens the mobile camera or camera/photo picker.
* After taking a picture, the app asks whether the user wants to rename the file.
* If renamed, the new name appears in Unassigned Documents.
* If not renamed, the image still appears in Unassigned Documents with a sensible default name.
* Desktop Unassigned Documents behavior still works.
* Existing drag/drop or file upload behavior is not broken.

**Regression risks to check**

* Desktop upload/dropzone behavior
* Mobile layout overflow or overlapping controls
* iPhone Safari camera capture
* Android Chrome camera capture
* Filename handling with spaces and special characters
* Upload progress/error states
* Assigning a newly captured unassigned document to a case/order afterward

### Debug Analysis

The most likely issue is not a backend problem. This sounds like a responsive UI gap: desktop has the Unassigned Documents window and upload flow, while mobile either hides that component or routes uploads through a different flow.

The best fix is to reuse the desktop Unassigned Documents component and connect mobile camera capture into the same queue. That avoids having desktop and mobile documents behave differently.

Alternate things Replit should check:

* The mobile layout may intentionally hide the desktop side panel at small breakpoints.
* The upload component may accept normal files but not use mobile camera capture attributes.
* The rename flow may already exist elsewhere and just needs to be called after capture.
* The unassigned document list may be tied to drag/drop events only, so mobile file input events need to feed into the same handler.

Confidence: medium-high on the product behavior and fix direction, but exact file names need Replit to inspect the repo. I found the likely app context from the LabTrax App Store listing, which describes adding pictures/videos/notes for labs: [https://apps.apple.com/us/app/labtrax/id6760672646](https://apps.apple.com/us/app/labtrax/id6760672646)
