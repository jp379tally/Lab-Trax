# Mobile AI Reader Intake — Desktop Workflow Parity

## What & Why

The mobile app is now close to beta quality. The next major feature is adding the AI Reader intake workflow back into the rebuilt mobile app.

This must not be a restoration of the old mobile AI Reader logic.

The mobile AI Reader must be rebuilt from the desktop/web intake workflow and must preserve every protected mobile beta workflow already confirmed working.

The AI Reader is a mobile hardware adapter:

* camera/document capture happens on mobile
* AI extraction happens through the canonical API
* case creation follows the same desktop/web workflow
* invoice generation follows the same desktop/web workflow
* files/history/attachments follow the same desktop/web workflow
* barcode/pan assignment writes to the same canonical case data

No local-only saves.
No `/api/legacy/cases`.
No duplicate invoices.
No regression of existing mobile beta functionality.

## User Workflow

The desired workflow is:

1. User opens mobile AI Reader.
2. User scans one or more dental prescriptions.
3. App detects prescription/letter boundaries automatically, similar to OneDrive.
4. Automatic shutter defaults OFF.
5. User can toggle automatic shutter ON.
6. Manual capture remains available at all times.
7. AI reads the prescription and extracts:

   * dentist/provider name
   * patient name
   * material
   * shade
   * tooth numbers
   * restoration type
   * due date if present
   * special instructions
   * any prescription notes
8. Multiple scanned prescription pages are combined and saved as a PDF.
9. The PDF is attached to the created case in the Files section.
10. The same PDF appears in case history as an attachment/history event with a tappable thumbnail/link.
11. Extracted fields prefill the same case-creation workflow used by desktop/web.
12. App detects possible duplicates or remake cases using the same rules/workflow as desktop/web.
13. User confirms:

* new case
* duplicate
* remake
* not a remake
* link to prior case if applicable

14. Case is created through canonical `POST /api/cases`.
15. Invoice is generated immediately through the same desktop/web invoice generation workflow.
16. User is prompted to scan a physical case pan barcode.
17. Barcode scan assigns that case pan/barcode to the canonical case.
18. Barcode can later be used to locate the case throughout the lab.
19. After barcode assignment, app asks:
    “Print case label?”
20. If yes, print label using the same desktop/web case-label format.
21. If no, finish workflow.
22. End state:

* case exists in production
* invoice exists once
* prescription PDF is attached
* prescription PDF is visible in history
* pan barcode is assigned
* case label was offered
* desktop/web shows the same data

### Capture

* Add mobile AI Reader screen.
* Support manual capture.
* Support automatic prescription/letter detection.
* Auto-shutter must default OFF.
* User can toggle auto-shutter ON/OFF.
* Support multiple prescription pages in one intake session.

### PDF Creation

* Combine captured prescription images into a single PDF.
* Attach the PDF to the canonical case.
* PDF must appear in:

  * Files section
  * Case history section
* History event must include a tappable thumbnail/link like desktop.

### AI Extraction

Use the canonical AI prescription analysis endpoint/workflow.

Extract:

* dentist/provider
* patient
* material
* shade
* tooth numbers
* restoration type
* due date
* notes/special instructions

Do not create a mobile-only parsing model.

### Desktop Workflow Parity

Case creation must follow the desktop/web workflow.

This includes:

* provider/practice matching
* unknown provider/practice handling
* duplicate detection
* remake detection
* restoration generation
* invoice generation
* attachment/history creation

If desktop prompts the user for a decision, mobile must prompt the user for the equivalent decision.

### Unknown Provider / Practice

If provider/practice is unknown:

* mobile must follow the same desktop/web provider creation/linking workflow
* do not show a dead-end “ask admin” message
* create/link provider canonically
* register alias if desktop does
* continue case creation
