### Replit Build Brief

**Objective:** Fix Customer Center so a newly added practice appears immediately and reliably after using **Add practice**.

**User-visible problem:** In Customer Center, adding a new dental practice appears to save, but the new practice does not show in the Customer Center table afterward. Screenshot shows Customer Center at `8 of 10`, with `Show archived (2)` unchecked, which suggests the page has more org records than it renders.

**Likely root cause:** The current Customer Center is `artifacts/labtrax-desktop/src/pages/accounts.tsx`, not the older `customer-center.tsx`. It fetches:

```ts
["organizations", { includeArchived: true, includeLabPractices: true }]
GET /organizations?includeArchived=true&includeLabPractices=true
```

Then it renders only orgs where:

```ts
o.type === "provider" || o.type === "lab"
```

and hides archived rows unless `Show archived` is checked.

The add-practice dialog lives in `artifacts/labtrax-desktop/src/pages/practices.tsx` as `AddPracticeDialog`. It posts to:

```ts
POST /organizations
```

with `type: "provider"` and `parentLabOrganizationId`.

The backend create path in `artifacts/api-server/src/routes/organizations.ts` looks mostly correct: it creates provider orgs under a parent lab and returns the new org. The likely failure is one of these:

1. The new provider is saved but the active Customer Center organizations query is not refetching/updating correctly after create.
2. The new provider is being saved with the wrong `type`, missing/wrong `parentLabOrganizationId`, or a non-null `deletedAt`, so Customer Center filters it out.
3. The backend list endpoint `GET /organizations?includeArchived=true&includeLabPractices=true` does not return the newly created provider for the active lab/admin user, even though `POST /organizations` succeeded.
4. The UI closes the dialog before the active accounts query has been updated, making the page appear stale until refresh or navigation.

**Relevant code areas:**

* `artifacts/labtrax-desktop/src/pages/accounts.tsx`

  * Main Customer Center page.
  * Fetches organizations with `includeArchived=true&includeLabPractices=true`.
  * Filters/render practices and lab orgs.
  * Opens `AddPracticeDialog`.

* `artifacts/labtrax-desktop/src/pages/practices.tsx`

  * Contains `AddPracticeDialog`.
  * Creates provider practice with `POST /organizations`.
  * Invalidates `["organizations"]` on success.

* `artifacts/api-server/src/routes/organizations.ts`

  * `POST /organizations`
  * `GET /organizations`
  * Parent lab resolution and `includeLabPractices` behavior.

* `artifacts/api-server/src/routes/organizations.test.ts`

  * Already has provider creation tests.
  * Needs a regression test proving the created provider appears in the `includeLabPractices` list response.

### Debug Analysis

Primary hypothesis: the practice creation path succeeds, but the Customer Center page does not refresh or reconcile the exact organizations query used by `accounts.tsx`, or the backend `includeLabPractices` list response does not include the newly created provider.

Strongest evidence: the app now routes Customer Center to `/accounts`, where `accounts.tsx` fetches `includeArchived=true&includeLabPractices=true`. The add dialog is reused from `practices.tsx` and invalidates organization queries after creation. The backend has tests for provider creation, but I did not see a specific regression test proving a newly created provider appears in the `includeLabPractices` list response.

Alternate hypotheses:

* The new record is saved with `deletedAt` populated or `isActive` false and is being hidden.
* The new record is saved under the wrong `parentLabOrganizationId`, so the current lab admin cannot see it.
* A persisted search/filter/session state is hiding it, though the screenshot search box appears empty and `With open cases only` is unchecked.

Why this fix path is best: it verifies the contract end-to-end instead of patching the display. The right invariant is simple: `POST /organizations` creates a provider practice under the active lab, and the Customer Center list query must include that provider immediately and after refresh.

### Step-by-Step Fix Plan

1. Add a uniquely named test practice and capture the POST response.
2. Inspect the returned org fields: `type`, `parentLabOrganizationId`, `deletedAt`, `name`, `displayName`.
3. Call `GET /organizations?includeArchived=true&includeLabPractices=true` as the same user.

5. If the org is present, fix `AddPracticeDialog`/`accounts.tsx` query cache refresh.
6. Add the backend regression test for create-then-list.
7. Add frontend coverage if feasible.
8. Run organization tests, desktop tests, and typecheck.
9. Manually verify Customer Center shows the newly added practice immediately and after refresh.
