This screenshot shows a **broken Windows taskbar shortcut**, not a normal LabTrax app error.

Windows is saying the pinned shortcut points to `LabTrax.exe`, but that file was moved, renamed, deleted, or replaced by an update/reinstall. That is why opening from the taskbar fails.

Immediate fix on that computer:

1. Click **No** on that dialog.
2. Unpin the broken LabTrax icon from the taskbar.
3. Open LabTrax from the working source: Start Menu, desktop icon, browser/PWA install, or the newest installer.
4. Once it opens successfully, right-click the running LabTrax icon and choose **Pin to taskbar**.

But the real regression fix is on the build/install side.

### Replit Build Brief

**Objective**
Fix the LabTrax desktop/taskbar launch regression so Windows shortcuts keep working after updates, rebuilds, or reinstalls.

**User-visible problem**
Pinned taskbar shortcut fails with:

```text
Problem with Shortcut
The item 'LabTrax.exe' that this shortcut refers to has been changed or moved,
so this shortcut will no longer work properly.
```

**Likely root cause**
The desktop build/update process changed or removed the installed executable path. Common causes:

* `LabTrax.exe` was renamed.
* install directory changed between builds.
* Electron/Tauri/native wrapper app ID or product name changed.
* installer is not cleaning/replacing old shortcuts.
* taskbar/start menu shortcuts point to a versioned or temporary path instead of a stable install path.
* previous desktop/PWA shortcut fix was overwritten by a later build.

**Implementation plan**

1. Identify how LabTrax desktop is packaged: Electron, Tauri, PWA shortcut, or Replit desktop wrapper.
2. Lock the app identity to stable values:

   * product name: `LabTrax`
   * executable name: `LabTrax.exe`

3. Ensure the installer always installs to a stable path, not a changing build/version folder.
4. Ensure updates replace the existing executable instead of creating a new app identity.
5. Ensure installer recreates Start Menu/Desktop shortcuts pointing to the current executable.
6. Add a release checklist/test that verifies:

   * Start Menu shortcut works
   * desktop shortcut works
   * pinned taskbar icon works after update
   * old shortcut is repaired or removed cleanly

**Guardrails**

* Do not rename the app, executable, package ID, or install folder unless absolutely required.
* Do not produce shortcuts to temporary build output folders.
* Do not let each build generate a new Windows app identity.
* Keep desktop launch behavior stable across updates.

**Acceptance criteria**

* Fresh install opens LabTrax from Start Menu.
* User can pin LabTrax to the taskbar and reopen it.
* After updating/redeploying, the pinned taskbar shortcut still opens LabTrax.
* No Windows “Problem with Shortcut” dialog appears.
* Old/broken shortcut is removed or repaired during install/update.

### Debug Analysis

The screenshot proves Windows still has a shortcut to `LabTrax.exe`, but that file is no longer at the shortcut target. That usually means the app was updated or rebuilt in a way that changed the executable path or app identity.

This kind of regression keeps coming back when the release process does not have a locked desktop identity and a shortcut test. The previous fix may have repaired one install or one shortcut, but the build system likely still allows a later release to rename, relocate, or regenerate the desktop executable.

Best fix: stabilize the desktop app identity and installer behavior, then add a release check so this cannot silently regress again.
