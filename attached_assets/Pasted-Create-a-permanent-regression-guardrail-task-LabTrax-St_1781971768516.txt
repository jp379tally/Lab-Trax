# LabTrax Stable Beta Regression Guardrails

## What & Why

LabTrax desktop and mobile are currently in a good working state. The biggest remaining uncertainty is backup/restore reliability, but the core desktop/mobile workflows are now working well enough that they must be protected permanently.

From this point forward, every future feature, bug fix, refactor, AI Reader change, mobile change, desktop change, billing change, auth change, or database migration must prove it does not regress the protected workflows.

## Rule

If a workflow is currently working in desktop or mobile, it becomes protected.

No future change may merge, publish, or trigger TestFlight/Desktop release if it breaks a protected workflow.

## Backup/Restore Special Rule

Backup and restore is business-critical.

No schema migration, auth change, organization change, case change, invoice change, or file/attachment change may merge unless backup/restore integrity tests pass or the change is explicitly marked as not touching persistent data.

Backup/restore failures block release.

## Regression Test Policy

When a bug is fixed:

1. Add a regression test that fails before the fix.
2. Add it to the protected workflow map if it affects core business data.
3. Keep that test permanently.
4. Do not delete or skip protected tests without explicit approval.

## Build Policy

EAS/TestFlight builds are manual only.

Desktop release publishing is manual only.

No automatic paid build or release should run without explicit approval.

## Acceptance

This task is complete when:

1. `REGRESSION_GUARDRAILS.md` is updated.
2. Protected workflow matrix exists.
3. Backup/restore is listed as a blocking protected workflow.
4. Pre-release checklist exists.
5. Manual smoke checklist exists.
6. CI/test commands are documented.
7. Any missing tests are proposed as follow-up tasks.
8. No code feature changes are bundled into this guardrail task.
