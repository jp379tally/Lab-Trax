### Replit Build Brief

**Objective**
Fix the provider portal so Dr. Craig can see SDR1 cases assigned to her actual SDR1 account number, and update the account-number model so one provider can work with multiple labs without cross-lab confusion.

**User-visible problem**
The provider portal shows:

```text
My Cases
drcraig · P-2026-1
You have no assigned cases yet.
```

But Dr. Craig has cases on file in **Lab Environment SDR1** under account number:

```text
2603-BC-1
```

The `P-2026-1` value shown under her name is not her lab account number and should not be used as the provider-facing account identifier.

**Likely root cause**
The portal is filtering cases by the provider portal user ID, internal provider ID, or generated `P-2026-1` identifier instead of resolving Dr. Craig to her **lab-specific provider/customer account** for SDR1.

The app likely has two concepts mixed together:

```text
Provider identity: Dr. Craig as a person/login
Lab-provider account: Dr. Craig's account relationship with SDR1
```

Cases should be visible through the lab-provider account relationship, not just the portal profile ID.

Likely code areas:


**Implementation plan**

1. Find what `P-2026-1` represents.

   * It is probably an internal provider profile number.
   * Do not display it as the provider’s lab account number.

2. Fix the provider portal case query.

   * Resolve logged-in provider to the active lab relationship.
   * For SDR1, Dr. Craig should resolve to account `2603-BC-1`.
   * Query cases by `labId + doctor/provider account relationship`, not by `P-2026-1`.

3. Add or formalize a lab-provider account relationship table/model if missing:

```text
providers
- id
- name
- email / login identity

labs
- id
- acronym, e.g. SDR1

lab_provider_accounts
- id
- providerId
- labId
- accountNumber
- legacyAccountNumber
- displayAccountNumber
- status
```

4. Cases should reference the lab-specific provider account where possible:

```text
case.labId
case.providerId or doctorId
case.labProviderAccountId
```

5. Account number rule going forward:

   * New account numbers should be lab-scoped.
   * Format should begin with the lab acronym.
   * Example:

```text
SDR1-2603-BC-1
ABC-1042-CR-1
```

6. Preserve existing account numbers as aliases.

   * Dr. Craig should still be findable by `2603-BC-1`.
   * Canonical display can become `SDR1-2603-BC-1`, but do not break existing login or existing cases.

7. Provider portal should show the correct lab-specific account number:

   * If logged into SDR1 context: show `SDR1-2603-BC-1` or `2603-BC-1`, whichever is selected as canonical.
   * Do not show `P-2026-1` unless labeled as an internal provider ID, and preferably hide it.

**Guardrails**

* Do not expose one lab’s account number to another lab.
* Do not make provider account numbers globally unique across all labs.
* Do not merge all Dr. Craig records blindly by name alone.
* Use lab scope for all provider portal case queries.
* Keep existing account numbers working during migration.
* Avoid breaking current case assignments.

**Acceptance criteria**

* Dr. Craig logs into the provider portal and sees SDR1 cases assigned to account `2603-BC-1`.
* The portal no longer shows `P-2026-1` as if it were her account number.
* If Dr. Craig works with another lab, she can have a separate account number for that lab.
* SDR1 cannot see or manage Dr. Craig’s account number from another lab.
* New provider account numbers start with the creating lab’s acronym.
* Existing cases linked to `2603-BC-1` remain visible after the fix.

### Debug Analysis

The provider portal is probably using the wrong identifier. The screenshot shows `drcraig · P-2026-1`, while the real SDR1 account number is `2603-BC-1`. If the portal query is filtering cases by `P-2026-1`, it will return zero cases even though SDR1 has cases for Dr. Craig.

The deeper design issue is that a doctor can work with multiple labs. Dr. Craig should have one provider identity for login, but separate lab-specific account relationships. SDR1’s account number for her should not have to match another lab’s account number.

Best model:

```text
One provider identity
Many lab-specific provider accounts
Cases linked through the lab-specific account
```

That keeps each lab’s account numbering private and prevents cross-lab confusion.
