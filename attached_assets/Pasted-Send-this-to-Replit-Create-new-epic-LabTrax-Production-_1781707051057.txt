# LabTrax Production Intake Fixes — Barcode, Lab Slip, AI Reader Data, Bulk Locate, and Autocomplete

## What & Why

We have several production workflow issues and missing pieces that need to be fixed before continuing with larger new features.

Some are regressions, some are missing workflow requirements. These must be handled carefully with zero regression.

The main goal is:

When a case is entered from the desktop AI Reader / drag-and-drop intake flow, all critical production data must carry through to the case, lab slip, history, invoice, and mobile/desktop workflows.

No local-only saves.
No mobile-only workflow divergence.
No regression to existing mobile beta functionality.

---

## Issue 1 — Assign Case Pan Barcode During Desktop Intake

### Requirement

During the desktop AI Reader / drag-and-drop prescription review step, before submitting the case into production, the user must be able to manually enter the physical case pan barcode.

This field should appear in the same review screen where the user sees:

* shade
* tooth number
* doctor/provider
* material
* restoration type
* notes/special instructions
* due date

### Barcode Format

Barcode values are numeric strings and may start with zeros.

Examples:

* `00001`
* `00002`
* `00125`

Important:

* Store barcode as a string, not a number.
* Preserve leading zeros.
* Do not trim meaningful leading zeros.
* Validate allowed characters before saving.

### Acceptance

* User manually enters barcode during AI intake.
* Case is created.
* Barcode is saved to the canonical case.
* Barcode appears on desktop case detail.
* Barcode appears on mobile case detail.
* Barcode can later be used for locate/scan workflows.
* Leading zeros remain intact.

## Issue 2 — Mobile Bulk Locate Multiple Cases

### Requirement

On the mobile app, user must be able to select multiple cases like desktop and locate them all at once.

### Workflow

* User selects multiple cases from the mobile cases list.
* User taps Locate / Move / Update Location.
* User chooses one location/station from the same desktop location list.
* All selected cases are updated together.
* Desktop/web reflects those updates.

### Acceptance

* Select multiple cases on mobile.
* Choose location.
* All selected cases update on desktop/web.
* Case history records the location change where desktop does.
* Uses canonical API only.
* No local-only mobile updates.

## Issue 5 — AI Reader Data Must Populate Lab Slip

### Current Problem

When a case is entered through desktop AI Reader / drag-and-drop intake, critical production data is not appearing on the lab slip.

Missing or unreliable fields include:

* material
* shade
* restoration type
* notes/special instructions

This makes the lab slip useless for production because technicians cannot see the required fabrication information.

### Requirement

All data extracted or confirmed in the AI intake review must be saved into canonical case/restoration/notes fields and rendered on the lab slip.

### Fields That Must Carry Through

* patient name
* doctor/provider
* tooth number
* restoration type
* material
* shade
* due date
* special instructions
* notes
* case pan barcode if entered
* priority if applicable

### Acceptance

1. Drag/drop prescription on desktop.
2. AI extracts material, shade, restoration type, and notes.
3. User confirms/edits fields.
4. Case is put into production.
5. Print lab slip.
6. Lab slip shows:

   * material
   * shade
   * restoration type
   * tooth number
   * notes/special instructions
   * due date
   * barcode if entered
7. Same data appears in desktop case detail.
8. Same data appears in mobile case detail.

## Issue 7 — Material Autocomplete With Add New

### Requirement

Material fields must support autocomplete.

Example:

* User types `zir`
* Dropdown suggests `zirconia`
* User selects it

If the material does not exist:

* show `Add New Material`
* user can add the new material
* new material becomes available for future autocomplete

### Applies To

* case overview edit section
* AI Reader review screen
* restoration edit forms
* desktop
* mobile where the same field exists

### Acceptance

* Typing partial material shows suggestions.
* Selecting suggestion fills field.
* Add New creates material option.
* Newly added material appears in later suggestions.
* Material saves canonically and appears on lab slip.

---

## Issue 8 — Global Autocomplete / Add New Pattern

### Requirement

The autocomplete pattern should not be limited to materials. Across LabTrax, fields should autocomplete where they represent reusable entities or controlled vocabulary.

Autocomplete + Add New should apply to:

* material
* shade
* doctor name
* practice name
* provider organization
* restoration type where applicable
* other reusable production fields engineering identifies

### Behavior

* User types into field.
* Dropdown suggests existing values.
* If no match, show Add New.
* Add New creates or registers the value canonically.
* Future users can select it.

### Important

Do not create duplicate doctors/practices/materials if a canonical record already exists.

Use canonical APIs and shared desktop/mobile logic.
