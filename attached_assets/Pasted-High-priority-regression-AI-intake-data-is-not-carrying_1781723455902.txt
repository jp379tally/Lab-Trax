High-priority regression: AI intake data is not carrying through to Lab Slip / Invoice / Case Detail.

I personally confirmed the AI intake extracted the correct prescription information, but after creating the case, critical fields disappeared or changed on the Lab Slip.

Example from current case:

* Patient: Park Brittle
* Case: 26-51
* AI intake confirmed fields correctly
* After case creation, Lab Slip shows:

  * Due date: —
  * Tooth #: —
  * Shade: —
  * Material: —
  * Restorative type: Other
  * Notes: No notes

This is wrong.

If the AI reads the prescription and the prescription says:

* due date = July 2
* tooth # = 19
* shade = A2
* material = zirconia
* restoration = crown

then the created case, Lab Slip, Invoice, Case Detail, and Mobile Case Detail must all show that same information.

## Required rule

The Lab Slip and Invoice must derive from the same canonical case/restoration data created from the AI intake review.

There must not be separate field mappings where:

* AI intake shows one thing
* created case stores something else
* Lab Slip reads from a different/empty field
* Invoice reads from a different/empty field

## Required investigation

Trace the full AI intake pipeline:

1. AI analysis result payload.
2. Review screen state.
3. User-confirmed values before submit.
4. Case creation payload.
5. Restoration creation payload.
6. Case record after creation.
7. Restoration records after creation.
8. Lab Slip render data source.
9. Invoice generation data source.
10. Mobile case detail data source.

Identify exactly where these fields are dropped or remapped:

* due date
* tooth number
* shade
* material
* restorative type
* notes / special instructions
* case pan barcode

## Required fix

When a case is created from AI intake:

* due date must save to canonical case due date
* tooth number must save to canonical restoration/tooth fields
* shade must save to canonical restoration/case shade fields
* material must save to canonical restoration material field
* restorative type must save to canonical restoration type field
* special instructions / notes must save to canonical case notes / Rx notes field used by Lab Slip
* barcode must save to canonical case pan barcode field if entered

Then:

* Lab Slip must render those fields from canonical data
* Invoice must render/generate from those fields
* Desktop case detail must show those fields
* Mobile case detail must show those fields
