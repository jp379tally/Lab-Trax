### Replit Build Brief

**Objective**
Fix a critical case-detail data integrity bug where the same case page shows correct RX Summary data but incorrect Patient & billing details from a different case.

**User-visible problem**
A case was created for:

* Patient: `Alba Hurtado`
* Doctor: `Edward Scott II / Dr. Scott`
* Teeth: `6-11`

The RX Summary correctly shows that information. But the editable Patient & billing details section shows unrelated data:

* Patient name: `Pam McGoff`
* Bill to: `Dr. Dalton`
* Teeth: `12`
* Shade: unrelated value

This means the case screen is mixing data from two different cases/doctors.

**Likely root cause**
The page is probably loading the correct case/RX summary by `caseId`, but the Patient & billing details form is being populated from stale state, cached form defaults, a previously selected invoice/case, or a non-case-scoped query. The form may not be resetting when the current case changes, or it may be using fallback invoice/customer data from the wrong record.

* `Rx Summary`

Then inspect:

* any route params such as `caseId`, `invoiceId`, `doctorId`, or `rxId`
* any Zustand/Redux/context/localStorage state used by the form

**Implementation plan**

1. Reproduce by opening the exact case for Alba Hurtado / Dr. Scott and confirming the mismatch.
2. Find where RX Summary gets its data. Treat that as the correct current case source.
3. Find where Patient & billing details gets its data.
4. Make the Patient & billing details form derive its defaults from the same current case/invoice record, scoped by the current `caseId` or `invoiceId`.
5. Remove or guard any fallback that pulls from the last selected case, last selected doctor, first invoice row, cached invoice draft, or global patient state.
6. Ensure the form resets whenever the route case/invoice id changes.
7. Add a defensive mismatch check: if the loaded billing details case id does not match the current case id, do not render the wrong data. Show a loading/error state instead.
8. Add regression coverage so a newly opened case cannot display patient/billing data from a previously opened case.

**Guardrails**

* Do not patch this by manually overwriting Alba’s data only.
* Do not hide the Patient & billing details section.
* Do not use patient name matching as the source of truth. Use stable IDs.
* Do not allow a billing/invoice record to populate the form unless its `caseId` or linked RX/case id matches the page’s current case.
* Be careful with saving: make sure clicking `Save changes` cannot write Pam/Dr. Dalton data onto Alba/Dr. Scott’s case.

**Acceptance criteria**

* Alba Hurtado’s case shows Alba Hurtado in both RX Summary and Patient & billing details.
* Bill To matches Dr. Scott / Edward Scott II, not Dr. Dalton.
* Teeth match the RX Summary teeth, not tooth `12`.
* Opening one case, then another case, never leaves stale patient/billing data behind.
* Saving changes only updates the currently opened case/invoice.
* If billing data is missing or mismatched, the app blocks save and shows a clear error instead of displaying another case’s data.

**Regression risks to check**

### Debug Analysis

**Probable root cause**
The RX Summary and Patient & billing details are being populated from different sources. RX Summary is using the correct case/RX record, while the editable billing form is likely using stale state or a wrongly scoped invoice/billing record.

**Supporting evidence**
The screenshot shows the contradiction on the same screen: RX Summary has Alba Hurtado and Edward Scott II, while Patient & billing details has Pam McGoff and Dr. Dalton. That is not a simple display typo. It means two different records are being merged into one UI.

**Alternate hypotheses worth checking**

* The database created the invoice with the wrong linked case id.
* The frontend is reusing previous form state after navigating between cases.
* The billing details query is keyed too broadly, such as by doctor, invoice status, or first open invoice.
* A draft restore from localStorage/sessionStorage is overriding the current case.
* The save/edit drawer is mounted once and not reset when a new case is selected.

**Why the recommended approach is best**
This needs a source-of-truth fix, not a cosmetic patch. The app should enforce that every patient/billing detail shown on a case page belongs to that exact case or invoice id. If the IDs do not match, the UI should block rendering and saving rather than risk corrupting another case.

Confidence: high that this is a stale-state or bad-record-scoping issue. Exact file names still need Replit to inspect in the project.
